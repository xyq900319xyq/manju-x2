## v1.1.5.17 (2026-07-11) — takeown + icacls 暴力解锁 Windows 文件 metadata 写权限(彻底修 v1.1.5.16 失败)

### 🟠 user 强烈反馈 v1.1.5.16 还是 v1.1.4
- user 反馈"没有用没有用,一点用也没有" + 截图(显示当前 v1.1.4,最新 v1.1.5.16)
- v1.1.5.13 / 5.14 / 5.15 / 5.16 连续 4 版都失败,user 已极度 frustrated
- **自查发现**:`installer/漫剧助手X-2.iss` 的 `CurStepChanged` **没真的写进** takeown + icacls 代码
  - v1.1.5.16 的 `X-2_v1.1.5.16_Setup.exe` 跑的是 v1.1.5.15 的代码(同步等 + RenameFile 兜底)
  - **没有** takeown + icacls 暴力解锁 metadata 写权限
  - Windows 文件被锁时 metadata 写权限被占,RenameFile 失败 → 装新文件时被锁跳过覆盖 → 还是 v1.1.4

### 真正根因(Windows 文件锁 + metadata 写权限)
- Windows 文件被锁(hermes.exe 子进程 / Defender 实时扫描 / 360 / explorer.exe 缩略图缓存)
- **不仅 del/rmdir 失败,连 RenameFile 改名也失败**——锁占 metadata 写权限
- v1.1.5.15 同步等子进程退 + RenameFile 兜底,锁释放不了 metadata 写权限,改名失败
- Inno Setup 装新文件时目标路径还有老 `_internal/`,静默跳过覆盖
- **必须**先 takeown + icacls 暴力夺权+赋权,del/rmdir/RenameFile 才能工作

### 修复(6 步,必须按顺序)
1. **`taskkill /F /IM`** 杀光 `漫剧助手X-2.exe` / `hermes.exe` / `python.exe` / `pythonw.exe`(`ewWaitUntilTerminated` 同步)
2. **`Sleep(5000)`** 等 Windows 文件句柄完全释放
3. **`takeown /F "{app}\_internal" /R /A /D Y`** 暴力夺整目录所有权给 administrators 组
4. **`takeown /F "{app}\漫剧助手X-2.exe" /A`** + **`icacls ... /grant administrators:F`** 同样处理 launcher EXE
5. **`icacls "{app}\_internal" /grant administrators:F /T /C /Q`** 暴力赋 administrators 完全控制权限
6. **`cmd /C del /F /Q`** + **`cmd /C rmdir /S /Q`** 才能真删(同步 + 二次检查 + RenameFile 改名兜底)

### 关键 takeaway
- **takeown 是关键**——没 takeown,administrators 也没 metadata 写权限
- **icacls 是关键**——没 icacls,owner 改了但 ACL 没改,还是写不了
- **必须两步都做**——只 takeown 不 icacls / 只 icacls 不 takeown 都不行

### 涉及代码
- `installer/漫剧助手X-2.iss` `[Code]` 段 `procedure CurStepChanged`
  - 加 `takeown /R /A /D Y` 暴力夺权
  - 加 `icacls /grant administrators:F /T` 暴力赋权
  - 之后再 del / rmdir

### 临时方案(等不及 v1.1.5.17 的 user)
- 任务管理器杀光所有 `漫剧助手X-2.exe` / `hermes.exe` / `python.exe` / `pythonw.exe`
- **手动以管理员身份**开 cmd:
  ```
  takeown /F "C:\漫剧助手X-2\_internal" /R /A /D Y
  icacls "C:\漫剧助手X-2\_internal" /grant administrators:F /T
  del /F /Q "C:\漫剧助手X-2\漫剧助手X-2.exe"
  rmdir /S /Q "C:\漫剧助手X-2\_internal"
  ```
- 装 v1.1.5.17 Setup.exe

## v1.1.5.15 (2026-07-11) — Exec 改同步等子进程退(Exec ewNoWait 异步)

### 🟠 user 反馈 v1.1.5.14 装完还是 v1.1.4
- user 反馈 "没有用,你依旧没有找到真正的问题"
- 截图显示"当前版本 v1.1.4,最新版本 v1.1.5.14,安装包大小 165.4 MB"
- 即使 v1.1.5.14 的 `CurStepChanged(ssInstall)` 整目录删,user 装完启动还是 v1.1.4

### 真正根因(Exec 默认异步!)
- v1.1.5.14 的 `Exec('cmd.exe', '/C del /F /Q ...', '', SW_HIDE, ewNoWait, ...)`
  `ewNoWait` 是 Inno Setup Pascal Exec 的**默认 wait 模式 = 异步**:
  - Inno Setup 调 Exec 立即返回,继续 `ssInstall` 阶段
  - 但 `cmd /C del /F /Q` 子进程还在跑,旧文件还没真删
  - 装新文件时目标路径还有旧文件(或被 cmd 进程持有)
  - Inno Setup 静默跳过覆盖 → EXE 仍是 v1.1.4
- `ewNoWait` 名字带 "No",很容易误以为"不等"但实际是 Inno Setup 的"no wait for ProcessHandle"

### 修复(`ewWaitUntilTerminated` 同步等)
1. **所有 `Exec` 改 `ewWaitUntilTerminated`**:同步等子进程退出,Inno Setup 才继续
2. **`Sleep` 加长**:3000 → 5000(taskkill 杀进程后),cmd del 后 2000,cmd rmdir 后 3000
3. **二次检查 + cmd 重试兜底**:
   - `del EXE` 后 `FileExists` 二次检查 → 还在就再 `del` 一次 → 还存在就 `RenameFile` 改名
   - `rmdir _internal/` 后 `DirExists` 二次检查 → 还在就再 `rmdir` 一次 → 还存在就 `RenameFile` 改名
4. **`ewWaitUntilTerminated` 改成所有 Exec**:taskkill / cmd del / cmd rmdir 全是同步

### 涉及代码
- `installer/漫剧助手X-2.iss` `[Code]` 段 `procedure CurStepChanged`
  - 所有 `ewNoWait` → `ewWaitUntilTerminated`
  - 加二次 `FileExists` / `DirExists` 检查
  - 加 cmd 重试 + `RenameFile` 兜底
  - `Sleep` 3000 → 5000

### 临时方案(等不及 v1.1.5.15 的 user)
- 任务管理器杀光所有 `漫剧助手X-2.exe` / `hermes.exe` / `python.exe` / `pythonw.exe`

## v1.1.5.11 (2026-07-10) — 一键更新去"取消"按钮(根治"用户取消"误触)

  没有触发安装流程。

### 真正根因(QProgressDialog race)
- v1.1.5.10 漏了**最关键的一环**:`QProgressDialog` 的**"取消"按钮**。

  2. `dlg.canceled` 信号触发
  3. `_downloader.cancel()` → `self._worker._cancel = True`
  4. worker 跑完当前 chunk 后,在 `while` 顶部 check 看到 True
  5. worker **删文件** + emit `error("用户取消")` + return

## v1.1.5.6 (2026-07-10) — manju 端主动探测 bash 路径,根因修

### 🟠 高严重修复
- **v1.1.5.5 装完 user 仍报"无法读取文件:Git Bash 未安装"**:
  v1.1.5.5 装了 PortableGit + 写 HKCU\Environment\HERMES_GIT_BASH_PATH,但
  user 覆盖装后启动 manju 仍然报。
  
根因:Windows 不会自动把 HKCU\Environment 的新值注入**已运行**的进程。
新值只对**新登录**或新 explorer 启动的进程生效。manju 进程是覆盖装后
启动的(从 .lnk 启动,继承 explorer 的 env block),env var 没生效。
hermes `_find_bash()` 第 1 步 `os.environ.get("HERMES_GIT_BASH_PATH")` 拿到空,
第 2-8 步的固定路径也都不存在(hermes 不知道 manju 装的 `<install_root>\PortableGit\`),
最终报"无法读取文件:Git Bash 未安装"。

修法(manju 端主动探测,不依赖系统 env var 加载):
- `core/generators.py` 加 `_find_bash_exe()` 函数,主动探测 8 个候选路径:
  1. `os.environ["HERMES_GIT_BASH_PATH"]` (用户命令行启动)
  2. `<manju_root>/PortableGit/bin/bash.exe` (v1.1.5.5 installer 装的,**主用**)
  3. `%LOCALAPPDATA%/hermes/git/bin/bash.exe` (hermes installer 装的 PortableGit)
  4. `%LOCALAPPDATA%/hermes/git/usr/bin/bash.exe` (hermes installer 装的 MinGit 32-bit)
  5. `shutil.which("bash")` (系统 PATH)
  6. `%ProgramFiles%/Git/bin/bash.exe` (system Git 64-bit)
  7. `%ProgramFiles(x86)%/Git/bin/bash.exe` (system Git 32-bit)
  8. `%LOCALAPPDATA%/Programs/Git/bin/bash.exe` (system Git 用户装)
- 加 `_ensure_hermes_bash_env(env)` 函数,找到 bash.exe 后**直接**塞
  `env["HERMES_GIT_BASH_PATH"]` + 把 bin/ 加到 PATH 头。覆盖原 env dict
  返回新 dict。
- 两处 env dict 构造点都加 `_ensure_hermes_bash_env(env)`:
  - `_build_hermes_call` (StoryboardTask / VideoPromptTask / 其他)
  - AssetExtractTask (不走 _build_hermes_call,独立构造 env)

辅助(让其他 app 也能立即看到 env var,无需 logout/login):
- ❌ Inno Setup 写 [Registry] 后**不**广播 WM_SETTINGCHANGE(原计划 [Code] 段
  调 `SendMessageTimeoutA(HWND_BROADCAST, WM_SETTINGCHANGE, 0, 'Environment', ...)`,
  build 试 4 次都报 "Unknown identifier" — Inno Setup 6 Pascal 不认 PChar /
  Pointer / PAnsiChar 任何字符串指针 cast,代码全删只留注释)。
