## v1.1.5.6 (2026-07-10) — manju 端主动探测 bash 路径,根因修

### 🟠 高严重修复
- **v1.1.5.5 装完 user 仍报"无法读取文件:Git Bash 未安装"**:
  v1.1.5.5 装了 PortableGit + 写 HKCU\Environment\HERMES_GIT_BASH_PATH,但
  user 覆盖装后启动 manju 仍然报。
  
根因:Windows 不会自动把 HKCU\Environment 的新值注入**已运行**的进程。
新值只对**新登录**或新 explorer 启动的进程生效。manju 进程是覆盖装后
启动的(从 .lnk 启动,继承 explorer 的 env block),env var 没生效。
hermes `_find_bash()` 第 1 步 `os.environ.get("HERMES_GIT_BASH_PATH")` 拿到空,
第 2-8 步的固定路径也都不存在(hermes 不知道 manju 装的 `<install_root>\PortableGit\`),
最终报"无法读取文件:Git Bash 未安装"。

修法(manju 端主动探测,不依赖系统 env var 加载):
- `core/generators.py` 加 `_find_bash_exe()` 函数,主动探测 8 个候选路径:
  1. `os.environ["HERMES_GIT_BASH_PATH"]` (用户命令行启动)
  2. `<manju_root>/PortableGit/bin/bash.exe` (v1.1.5.5 installer 装的,**主用**)
  3. `%LOCALAPPDATA%/hermes/git/bin/bash.exe` (hermes installer 装的 PortableGit)
  4. `%LOCALAPPDATA%/hermes/git/usr/bin/bash.exe` (hermes installer 装的 MinGit 32-bit)
  5. `shutil.which("bash")` (系统 PATH)
  6. `%ProgramFiles%/Git/bin/bash.exe` (system Git 64-bit)
  7. `%ProgramFiles(x86)%/Git/bin/bash.exe` (system Git 32-bit)
  8. `%LOCALAPPDATA%/Programs/Git/bin/bash.exe` (system Git 用户装)
- 加 `_ensure_hermes_bash_env(env)` 函数,找到 bash.exe 后**直接**塞
  `env["HERMES_GIT_BASH_PATH"]` + 把 bin/ 加到 PATH 头。覆盖原 env dict
  返回新 dict。
- 两处 env dict 构造点都加 `_ensure_hermes_bash_env(env)`:
  - `_build_hermes_call` (StoryboardTask / VideoPromptTask / 其他)
  - AssetExtractTask (不走 _build_hermes_call,独立构造 env)

辅助(让其他 app 也能立即看到 env var,无需 logout/login):
- ❌ Inno Setup 写 [Registry] 后**不**广播 WM_SETTINGCHANGE(原计划 [Code] 段
  调 `SendMessageTimeoutA(HWND_BROADCAST, WM_SETTINGCHANGE, 0, 'Environment', ...)`,
  build 试 4 次都报 "Unknown identifier" — Inno Setup 6 Pascal 不认 PChar /
  Pointer / PAnsiChar 任何字符串指针 cast,代码全删只留注释)。
- ✅ [Registry] HKCU\Environment\HERMES_GIT_BASH_PATH 仍写入(用户能在
  regedit 看到 + 下次新登录后生效,作为辅助手段)。

Installer 大小:跟 v1.1.5.5 一样(173 MB,没新增资源,只改 Python + Inno Setup 代码)

## v1.1.5.5 (2026-07-10) — 自带 Git Bash (MinGit) 装机,跟老 software 行为一致

### 🟠 高严重修复
- **生成 Seedance 提示词时 hermes 报 "无法读取文件:Git Bash 未安装"**:
  user EXE 装机(没装 Git Bash)触发。hermes 的 terminal 工具在 Windows 上
  调 `bash -c 'cat <file>'`,依赖 Git Bash (HERMES_GIT_BASH_PATH env var)。
  老 software D:\剧本分镜助手\ 装 hermes 时自带 PortableGit + 设
  HERMES_GIT_BASH_PATH (hermes install 脚本),老 software 没爆。
  v1.1.5.1 EXE 装机是**第一次**没装 Git Bash 的场景。

修法(user 明确要求 + 反对截断):manju install 脚本**自带 MinGit** 跟老 software 行为一致。

- `build_x2.py` 加 step 0.5:下载 MinGit-2.54.0-64-bit.zip (~10MB) → 解压到
  `installer/PortableGit/`,含 `bash + cat + sh + awk + sed + grep + curl`。
  缓存机制:已存在跳过下载。Python `zipfile` 解压不用 7z。
- `installer/漫剧助手X-2.iss` 加 `Source: installer\PortableGit\*` → `{app}\PortableGit`。
- `installer/漫剧助手X-2.iss` 加 `[Registry]` 段:HKCU\Environment\HERMES_GIT_BASH_PATH
  = `{app}\PortableGit\cmd\bash.exe`,hermes 端直接读。卸载时自动清 (`uninsdeletevalue`),
  写失败不阻塞 (`errorignore`)。
- v1.1.5.5 之前 `prompts.py` / `generators.py` 写的"截断"被 user 反对,**已撤回**,
  恢复原行为:长 storyboard/script 写 tmp file + 让 hermes cat,hermes 收完整内容。

未装 Git Bash:
- 之前 v1.1.5.1 ~ v1.1.5.4:broken,报 "无法读取文件:Git Bash 未安装"
- v1.1.5.5:✅ 装 MinGit 跟老 software 一致,长分镜/剧本 work

## v1.1.5.2 (2026-07-10) — 批量资产生图 SQLite 跨线程修复

### 🟠 高严重修复
- **批量资产生图失败**:`BatchAssetImageTask.run()` 在 task queue worker thread 调
  `self._db.list_assets/update_asset_image/update_asset_status`,但 `self.db` 的
  `sqlite3.Connection` 是 main thread 创建的(`main_window.py:118` → `open_db(ROOT)`),
  SQLite3 默认 `check_same_thread=True` → 跨线程报 "SQLite objects created in a
  thread can only be used in that same thread. The object was created in thread
  id 14720 and this is thread id 14960."
  修法:`core/migration.py open_db` 加 `check_same_thread=False`。SQLite 内部有
  mutex 保护并发安全,task queue 串行跑无 race。
  其他 5 类 task(`StoryboardTask` / `VideoPromptTask` / `AssetExtractTask` /
  `AssetImageTask` / `VideoTask`)通过 hermes 处理不直接读 db 不受影响。
